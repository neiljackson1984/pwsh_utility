    +1 to Vasil.

    You can do something like this:

    try 
{ $var = Get-AzureADTenantDetail } 

catch [Microsoft.Open.Azure.AD.CommonLibrary.AadNeedAuthenticationException] 
{ Write-Host "You're not connected."; Connect-AzureAD -credential $cred}

    You can find out the full name of the exception you're trying to catch by referencing $Error.Exception.GetType().FullName.

    This worked for me in Powershell:

    if($azureConnection.Account -eq $null){
    $azureConnection = Connect-AzureAD
}

    Basically, on initial run, it'll set the connection information to the $azureConnection variable. Then, when re-run, it'll find that the Account is populated, so it will skip the connection step. I hope this helps, even though it's almost a year after the initial question.

    I've also come to a similar question.  After a LOT of digging in the cmdlets .dll
    I found this to be the "most correct" way.

    This is the variable you should reference, as it is the variable the Azure cmdlets reference.
    [Microsoft.Open.Azure.AD.CommonLibrary.AzureSession]::AccessTokens

    if it is null or .count = 0  then it throws the exception.  

    Full bit of code:
    // Microsoft.Open.Azure.AD.CommonLibrary.AzureSession
    public staticvoid ClientAuthenticated()
    {
    if (AzureSession.AccessTokens == null || AzureSession.AccessTokens.Count == 0)
        {
    thrownew AadNeedAuthenticationException(Messages.NotAuthenticated, null);
        }
    }
